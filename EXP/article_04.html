為什麼只做「提示詞產生器」？
關於 AI 塔羅占卜的底層邏輯與產品哲學

Product Philosophy Prompt Engineering Generative AI
2026.02.19 BY GHOST_1010

部署「塔羅 Prompt 煉金術師」這項專案後，最常被問到的問題是：「為什麼不直接串接 API，做成一個一鍵生成結果的 AI 塔羅牌算命機？」

這其實不僅僅是技術路線的選擇，更牽涉到生成式 AI (Generative AI) 時代下，兩種截然不同的產品架構思路，以及對 API 算力成本的現實考量。

# 封閉式預言機 vs. 開放式提示詞框架

市面上主流的 AI 塔羅占卜服務，多半採用「黑箱直出」模式：使用者輸入問題、抽牌，系統直接回傳一段長篇大論的解讀。而我的專案，則是專注於「提示詞工程 (Prompt Engineering)」，負責生成一段高度封裝的環境指令，讓使用者將其部署到自己習慣的 ChatGPT 或 Claude 介面中。

這兩種機制沒有絕對的優劣，而是取決於產品定位與受眾的痛點：

封閉式解析平台 (黑箱模式)

  • 降低認知門檻：麻瓜友善，隨開即用，無需理解 AI 運作邏輯，也不需要自行切換視窗。
  • 算力成本轉嫁：開發者需負擔每一筆 API 請求 (Token) 費用。為維持營運，平台通常需建立付費牆 (Paywall) 限制追問次數，或置入大量廣告。
  • 缺乏上下文深度：受限於架構，多為「一問一答」的線性腳本，難以針對問卜者當下的幽微情緒進行動態收斂與深度追問。

環境指令產生器 (開源框架)

  • 算力自主化 (無限追問)：運用使用者自身的 Token (如免費版 ChatGPT) 進行推論，打破 API 呼叫次數的限制，實現零成本的深度對話。
  • 賦權使用者：將「塔羅師」的解讀權限交還給使用者慣用的 LLM，從單向的預言機轉變為具備記憶的雙向心理諮商空間。
  • 存在操作壁壘：使用者需具備基本的 Prompt 概念，願意多花五秒鐘手動完成指令的複製與環境切換。

# 塔羅的底層邏輯：並非線性程式碼，而是動態心理投射

選擇打造「把提示詞直接帶走」的產生器，核心在於我對塔羅占卜本質的拆解。

塔羅牌並非純粹理性的邏輯工具。它不是「A 牌 + 正位 = B 結果」的靜態程式碼。塔羅牌的精準度，極度仰賴當下的語境 (Context)、問卜者的心理投射，以及最重要的——「對話中的資訊聚焦」。

一張牌的深層意涵，往往無法單憑字面定義來窮盡。它必須依賴塔羅師的經驗庫，並在與問卜者一問一答的交互過程中，如剝洋蔥般逐步解構。這是一種高度動態的資訊流動。如果將其做成單向的一鍵解答平台，AI 往往只能輸出正確卻缺乏溫度的「維基百科式」解析，這便抹殺了塔羅牌中透過對話釐清思緒的核心價值。

# 角色解耦：將「解析引擎」交還 AI，專注於「環境封裝」

為了解決單向輸出的痛點，我的架構策略是：「將『塔羅老師』這個解析引擎 (Parser)，完整地交給更強大的生成式 AI；而我，只負責編寫精準的『環境變數與人設指令』。」

透過深度優化的 ChatGPT 詠唱技巧，我賦予了 AI 資深塔羅師的視角與同理心，嚴格限制其「套版直出」的行為，並強制要求它在給出最終建議前，必須先向使用者「反問」以收集更多參數。當使用者將這段指令輸入 AI 時，啟動的將不再是一份靜態的報告，而是一個隨時待命、具備上下文記憶、且能協助你收斂問題的心靈導師模型。

「每個人都有其高度適配的 AI 工具與內省節奏。與其打造一台受限於商業成本、僅能輸出標準答案的算命終端機，我更傾向提供一把經過精密計算的鑰匙，讓使用者掌握主動權，開啟專屬於自己的深度對話空間。」

這便是「塔羅 Prompt 煉金術師」的產品哲學。封閉式平台或許降低了多數人仰望星空的門檻，但我希望提供的，是一副能讓你在龐大的潛意識星海中，自由導航且不受限制的開源星圖指南針。
